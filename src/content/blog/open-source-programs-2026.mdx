## Why Open Source Matters

Before the list: why bother?

- **Real portfolio** — GitHub contributions > personal projects nobody sees
- **Production experience** — Work on codebases with millions of users
- **Network effects** — Maintainers remember good contributors
- **Hiring signal** — Companies actively recruit from GSoC, Outreachy, etc.
- **Paid opportunities** — Many programs offer $3,000-$6,000+ stipends

## Tier 1: Flagship Programs (Competitive, High Reward)

These are the most recognized programs. Competitive but worth the effort.

### Google Summer of Code (GSoC)
**Stipend:** $1,500-$6,600 depending on location  
**Timeline:** March-September annually  
**Who:** Students and early-career developers worldwide  
**Link:** [summerofcode.withgoogle.com](https://summerofcode.withgoogle.com/)

The gold standard. Work with established open source orgs (Mozilla, TensorFlow, Django, etc.) for 10-22 weeks. Excellent mentorship, strong resume signal.

**Pro tip:** Start contributing to your target org 2-3 months before applications open.

### Outreachy
**Stipend:** $7,000 total  
**Timeline:** Two cohorts per year (May and December)  
**Who:** People from underrepresented groups in tech  
**Link:** [outreachy.org](https://www.outreachy.org/)

Three-month paid internships with strong mentorship. Explicitly supports people facing systemic barriers. Application includes a contribution period.

### MLH Fellowship
**Stipend:** Varies by track  
**Timeline:** Multiple cohorts year-round  
**Who:** Students and recent graduates  
**Link:** [fellowship.mlh.io](https://fellowship.mlh.io/)

Remote fellowship with three tracks: Open Source, Production Engineering, and Explorer. Work on real projects with industry mentors.

## How to Actually Get Accepted

Programs are competitive. Here's what works:

**1. Start contributing before you apply**  
Maintainers remember names. Make 3-5 quality contributions to your target org before applications open.

**2. Start small**
- Fix typos in documentation
- Update broken links
- Add missing tests
- Improve error messages

These build trust before you tackle big features.

**3. Read everything**
- README files
- CONTRIBUTING.md
- Existing issues and discussions
- Code of conduct

Most rejections come from not following project guidelines.

**4. Be patient with feedback**  
Code reviews can feel harsh. They're not personal. Maintainers want you to succeed.

**5. Write good commit messages**  
"Fixed bug" → Bad  
"Fix null pointer exception in user authentication when email is empty" → Good

## The 30-Day Quickstart

**Week 1:** Set up your GitHub profile. Add a bio, photo, and pinned repos.

**Week 2:** Find 3 beginner-friendly projects. Look for "good first issue" labels.

**Week 3:** Make your first contribution (documentation counts).

**Week 4:** Engage in discussions. Help others. Build relationships.

After 30 days, you'll have GitHub activity, at least one merged PR, and a feel for which communities you enjoy.
